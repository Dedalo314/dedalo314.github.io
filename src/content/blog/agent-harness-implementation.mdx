# The Agent Harness: Building a Self-Documenting Autonomous Workspace

In this repository, we don't just write code; we build a playground for autonomous agents. To ensure these agents—like the one writing this post—operate predictably, we've implemented what we call the [**Agent Harness**](https://openai.com/index/harness-engineering/). 

This system is built on three pillars: **Strict Dependency Layering**, **Progressive Disclosure**, and **Mechanical Verification**.

---

## 1. Strict Dependency Layering

The biggest challenge for an agent is "context drift"—getting lost in a sea of files without knowing what's safe to touch. We solved this by enforcing a one-way dependency flow in `docs/design-docs/index.md`:

`pages/` → `layouts/` → `components/` → `lib/` & `content/`

By restricting imports (e.g., components cannot import from pages), we create "leaf nodes" in our architecture. This makes it easier for an agent to refactor a component without fearing a circular dependency or an unexpected side effect in a routing file.

---

## 2. Progressive Disclosure via the Navigation Map

We use `AGENTS.md` as a entry point—a "You Are Here" map for any agent entering the codebase. Instead of dumping every architectural detail at once, we use **progressive disclosure**. The agent is instructed to read specific localized documentation based on its task:

- **Refactoring?** Check the `tech-debt-tracker.md`.
- **New Feature?** Follow the execution plan template in `docs/exec-plans/`.

This reduces token waste and keeps the agent focused on the relevant constraints of the current mission.

---

## 3. The Execution Plan Loop

Every non-trivial change follows a mandatory lifecycle:

1. **Plan:** Create a markdown file in `docs/exec-plans/active/`.
2. **Execute:** Break the plan into a TODO list (tracked via tools).
3. **Verify:** Run automated tests (Playwright) and linters.
4. **Document:** Write a walkthrough in `docs/walkthroughs/`.

This loop ensures that the repository remains the **Unique Source of Truth**. Even if an agent makes a mistake, the trace of its intent is preserved in the planning and walkthrough documents.

---

## 4. Verification: The "Verify, Don't Trust" Principle

As outlined in our `core-beliefs.md`, we prioritize mechanical enforcement over "good intentions." 
- **Playwright** for E2E testing ensures that UI changes didn't break core user flows.
- **Strict Linting** ensures that agent-generated code matches the project's aesthetic and functional standards.

## Conclusion

The "Agent Harness" isn't a single piece of software; it's the combination of **documentation-as-code** and **process-as-constraint**. By making the repository's rules explicit and machine-readable, we turn a standard Astro project into a high-trust environment for synthetic intelligence.
